1. SYSTEM ARCHITECTURE

HOSTHAVEN uses a Client-Server architecture:

User Browser
   |
   | HTTPS / REST JSON
   V
Frontend (React + Tailwind)
   |
   | REST API / Webhook
   V
Backend (Express.js + Node)
   |
   +--> MongoDB Atlas
   +--> Cloudinary (Media)
   +--> Razorpay (Payments)
   +--> Google Maps (Location)

Key architectural patterns:

Stateless backend with JWT session tokens

Token validation with Firebase for OTP users

Role-based middleware (User / Vendor / Admin)

REST APIs with versioning (/v1/...)

Webhook listener for Razorpay payments

3. USER STORIES & ACCEPTANCE CRITERIA

3.1 CUSTOMER

Story: As a user, I want to log in with email/password so I can access my account.
Acceptance: Credentials validated, JWT issued.

Story: As a user, I want to search hotels/homes by city + dates.
Acceptance: List shows accurate availability.

Story: As a user, I want to see hotel video before booking.
Acceptance: Video player loads and plays without errors.

Story: As a user, I want to pay via Razorpay.
Acceptance: Payment success triggers booking.


---

3.2 VENDOR

Story: As a vendor, I want to log in using email/password.
Acceptance: Password must be hashed and validated.

Story: As a vendor, I want to manage my hotel inventory.
Acceptance: Inventory updates reflect in real time.


---

3.3 ADMIN

Story: As an admin, I want to manage services/temples/homes.
Acceptance: Admin UI updates DB.

Story: As an admin, I want to view support tickets.
Acceptance: Support list shows statuses.

5. AUTHENTICATION & AUTHORIZATION

5.1 USERS (Email/Password)

User submits credentials

Backend validates credentials

Generate session JWT with role user

Optional: Google OAuth login


JWT payload:

{ userId, role: "user", iat, exp }


---

5.2 ADMIN & VENDOR (Email/Pass)

Use bcrypt for hashing

Issue JWT with roles


{ userId, role: "admin"|"vendor" }


---

5.3 AUTH MIDDLEWARE

Verify JWT

Attach user object to request

Role guard:

Admin only

Vendor only (and matching hotelId)

8. REAL-TIME INVENTORY LOGIC

Lock Process

1. Check availability on all days


2. Insert “locked” record


3. Set TTL of 5–10 minutes


4. On payment success -> decrement available


5. On failure -> release lock
